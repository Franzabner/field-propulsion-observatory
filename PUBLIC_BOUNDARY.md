# Public Boundary

## Public-Safe Content

| Content | Public posture |
| --- | --- |
| Documentation | Public-safe if kept within evidence, metadata, simulation, and boundary language. |
| Schemas | Public-safe structural definitions for synthetic and reviewed records. |
| Examples | Synthetic only. No real named cases. |
| Simulation engine | Lightweight public-safe demonstration logic only. |
| Templates | Public-safe review and report templates. |
| Banners | Non-sensational technical visuals only. |

## Not Public

- Private DGX or rack topology.
- Private lab notebooks.
- Sealed YOSO-YAi LLC IP.
- Customer, donor, student, or volunteer data.
- Real classified cases or disputed real-event conclusions.
- Tokens, secrets, infrastructure credentials, or authentication records.
- Unreviewed Hugging Face artifacts.
- Live Upwork offer copy.
- Real pilot, whistleblower, agency-personnel, or disputed-event references used as evidence.
- Real case accusations against people, organizations, agencies, or programs.
- Fake claims that a dataset, model, Space, report, or release exists when it has not been created and reviewed.

## Blocked Public Frames

The repository must block publication language that says or implies:

- aliens confirmed;
- anti-gravity proven;
- zero-point propulsion proven;
- recovered craft or recovered materials verified;
- classified program claims verified;
- free energy demonstrated;
- government coverup framing;
- real case accusations;
- named real pilot, whistleblower, or disputed-event conclusions;
- Hugging Face dataset, model, Space, or report released when no reviewed artifact exists.

## Boundary Rule

Public means content can be published without exposing secrets, sealed IP, donor-sensitive information, customer-sensitive information, unreleased operational plans, or unsupported propulsion claims.

If a future contribution introduces a real case, real sensor data, live infrastructure detail, or external artifact release plan, it requires human review before publication.

Generated ledgers under `sim/outputs/*.ledger.json` are working outputs. They must not be staged as source unless a reviewer intentionally promotes a specific ledger as a synthetic example fixture.
